# CodePilot Review PRD

## 1. 项目概述

**项目名称**：CodePilot Review

**选题方向**：题目三：AI PR Review 助手

**项目定位**：面向开发者和代码评审者的 AI 辅助 Pull Request 评审工具。用户输入 GitHub PR 链接后，系统自动获取 PR 元数据、代码 diff 和相关上下文，通过 AI 工作流生成 PR 变更总结、风险代码识别、Review 建议和测试建议，帮助开发者提升代码评审效率与质量。

**一句话描述**：让开发者在几分钟内看懂 PR 变更重点、潜在风险和可执行的 Review 建议。

## 2. 背景与问题

开发者在日常代码评审中常遇到以下问题：

- PR 涉及文件较多时，评审者需要花大量时间理解变更意图。
- 仅阅读 diff 容易遗漏上下文，例如调用链、配置变更、测试覆盖和数据结构影响。
- Review 建议质量不稳定，容易出现泛泛而谈、缺少证据或无法落地的问题。
- 高风险问题如安全、异常处理、兼容性、数据处理和性能问题，需要有经验的评审者才能稳定发现。
- 评审过程缺少统一结构，PR 总结、风险判断和测试建议往往依赖个人习惯。

CodePilot Review 的目标不是替代人工 Review，而是作为评审前置助手，帮助开发者快速建立上下文、发现高风险点，并生成有证据支撑的建议。

## 3. 目标用户

### 3.1 核心用户

- 后端、前端、全栈开发者
- 需要频繁 Review PR 的团队成员
- 开源项目维护者
- 技术负责人或实习生导师

### 3.2 使用场景

- 合并 PR 前快速了解变更内容。
- 评审大改动 PR 前获取风险清单。
- 自查自己提交的 PR，提前发现问题。
- 生成结构化 PR Review 建议，辅助团队沟通。
- 作品演示时展示 AI 在代码理解、风险识别和工程落地中的能力。

## 4. 产品目标

### 4.1 功能目标

- 支持用户输入 GitHub PR URL。
- 自动解析 PR 所属仓库、PR 编号和基础元数据。
- 自动获取 PR diff、变更文件、增删行数和提交信息。
- 结合 diff 与相关代码上下文进行分析。
- 生成 PR 变更总结。
- 识别风险代码并按严重程度分类。
- 生成可执行的 Review 建议和测试建议。
- 支持流式展示分析进度。
- 支持模拟发布 Review 评论。
- 提供模型选择、上下文获取方式和未来扩展方向说明。

### 4.2 非功能目标

- 输出建议必须引用具体代码证据。
- 降低幻觉、误报和泛泛建议。
- 分析过程可观测，前端能展示当前节点进度。
- 支持 Demo 模式，避免依赖真实 GitHub 写权限。
- 主分支保持可运行状态。
- README、Demo 视频和提交记录满足比赛交付规范。

## 5. 范围边界

### 5.1 MVP 范围

- GitHub PR URL 解析。
- GitHub PR 元数据和 diff 获取。
- Mock GitHub Provider，用于无 Token 或演示环境。
- 基于 LangGraph 的多节点分析流程。
- PR 总结、风险识别、Review 建议和测试建议。
- FastAPI 后端接口。
- 前端 PR Review Dashboard。
- SSE 或 WebSocket 流式进度。
- 模拟发布 Review 评论。
- README、Demo 脚本、提交规范和最终检查清单。

### 5.2 暂不实现

- 真实向 GitHub 写入 Review 评论。
- GitLab、Gitee 等平台接入。
- 完整代码调用链静态分析。
- 企业级权限系统。
- 多用户协作空间。
- 长期历史 Review 数据统计。

## 6. 核心流程

1. 用户打开 CodePilot Review 工作台。
2. 用户输入 GitHub PR URL。
3. 系统解析 owner、repo 和 PR number。
4. 系统获取 PR 元数据、changed files 和 unified diff。
5. 系统获取变更文件内容和附近函数、类、测试文件等上下文。
6. LangGraph 工作流依次执行总结、风险分析、建议生成和自检。
7. 前端通过 SSE 或 WebSocket 展示节点进度和部分结果。
8. 用户查看总结、风险列表、Review 建议和测试建议。
9. 用户选择建议并生成 GitHub 风格的模拟评论预览。
10. 用户可根据报告手动完成真实 Review。

## 7. 系统架构

### 7.1 后端模块

- **API 层**：FastAPI，提供任务创建、PR 解析、任务状态、流式事件和报告查询接口。
- **GitHub Provider 层**：封装 GitHub API 与 Mock Provider，负责 PR 元数据、diff 和文件内容获取。
- **Context Collector**：提取变更文件上下文、附近函数或类、相关测试文件和配置文件。
- **Review Workflow**：使用 LangGraph 编排输入校验、PR 获取、上下文收集、总结、风险分析、建议生成和最终报告组装。
- **LLM Client**：封装模型调用、结构化输出解析、重试和降级逻辑。
- **Guardrail 模块**：进行证据校验、置信度过滤、重复建议去重和低质量建议剔除。
- **Task Store**：保存任务状态、节点进度和分析结果，MVP 可使用内存存储。

### 7.2 前端模块

- **PR 输入区**：输入 PR URL，显示解析结果和错误提示。
- **进度区**：展示当前分析阶段、节点状态和异常提示。
- **摘要区**：展示 PR 变更目标、模块分组和评审重点。
- **风险区**：按严重程度、风险类别、文件路径展示风险发现。
- **建议区**：展示 Review 评论、修复建议和测试建议。
- **模拟发布区**：展示 GitHub 风格的评论预览。

## 8. 模型与上下文策略

### 8.1 模型选择

- 快速模型用于 PR 概览总结和文件分组。
- 强推理模型用于风险识别、Review 建议和最终自检。
- 模型输出使用 Pydantic Schema 约束，减少格式不稳定问题。
- 当模型输出结构异常时，允许重试一次，并提供降级结果。

### 8.2 上下文获取

系统不仅分析 diff，还会补充以下上下文：

- PR 标题、描述、作者、分支和提交信息。
- changed files、增删行数和 unified diff。
- 变更文件当前内容。
- 修改代码附近函数或类定义。
- 相关测试文件、schema、API 路由和配置文件。
- 文件路径、行号、hunk 范围等证据标签。

### 8.3 误报与漏报控制

- 高风险和阻塞类建议必须包含具体代码证据。
- 每条风险发现需要包含严重程度、类别、原因和置信度。
- 默认隐藏低置信度发现。
- 增加最终自检节点，移除重复、缺少证据或过于宽泛的建议。
- 当上下文不足时，明确提示分析不完整，不强行给结论。

- 创建前端应用壳，包含单页 PR Review 工作台。
- 定义 GitHub Token、模型供应商、模型名称和请求超时等环境配置。
- 使用 Pydantic 定义 Review 任务请求和响应 Schema。
- README 包含本地开发启动说明。
- 后端测试通过。
- 前端类型检查通过。

### US-002 Parse GitHub PR input

**用户故事**：作为用户，我希望粘贴 GitHub PR URL 后系统能自动识别仓库、所有者和 PR 编号。

**优先级**：2

**验收标准**：

- 支持 `https://github.com/{owner}/{repo}/pull/{number}` 格式。
- 非法 URL 返回清晰校验提示。
- 将 owner、repo 和 pull request number 提取为内部类型模型。
- 分析开始前展示解析后的 PR 元数据预览。
- 后端测试通过。
- 前端类型检查通过。

### US-003 Fetch PR metadata and diff

**用户故事**：作为用户，我希望助手自动从 GitHub 获取 PR 变更，避免手动复制代码。

**优先级**：3

**验收标准**：

- 使用 GitHub API 获取 PR 标题、作者、base 分支、head 分支、变更文件、增删行数和 commit 数量。
- 获取 changed files 的 unified diff，并保留文件路径和行范围。
- 配置有效 GitHub Token 时支持私有仓库。
- 对缺少 Token、无权限、限流和 PR 不存在返回可操作错误。
- 增加 Mock GitHub Provider，用于演示和测试。
- 后端测试通过。

### US-004 Collect relevant code context

**用户故事**：作为评审者，我希望助手查看变更代码周边上下文，让 Review 建议不只依赖原始 diff。

**优先级**：4

**验收标准**：

- 可获取 changed files 的当前文件内容。
- 尽可能使用语言感知启发式规则提取修改 hunk 附近函数或类上下文。
- 通过路径模式识别相关测试、schema、API 路由和配置文件。
- 对每个文件设置确定性的 token 或字符预算。
- 为上下文附加来源标签，方便生成结果引用证据。
- 后端测试通过。

### US-005 Generate PR change summary

**用户故事**：作为评审者，我希望得到简洁 PR 总结，以快速理解变更内容和评审重点。

**优先级**：5

**验收标准**：

- 总结覆盖业务意图、关键变更模块和重要实现细节。
- 将变更文件按 backend、frontend、tests、config、docs 或 unknown 分组。
- 返回结构化 JSON，包含 overview、file groups 和 reviewer focus areas。
- 模型输出格式异常时重试一次，失败后返回降级摘要。
- 后端测试通过。

### US-006 Detect risky code changes

**用户故事**：作为评审者，我希望高风险代码被突出显示，以便优先关注正确性、安全性和可维护性问题。

**优先级**：6

**验收标准**：

- 覆盖 correctness、security、performance、compatibility、observability、data handling 和 missing tests 等风险类别。
- 每条发现标记 critical、high、medium 或 low 严重程度。
- 每条发现包含文件路径、可用行引用、证据、推理和置信度。
- 默认视图隐藏低于配置阈值的低置信度发现。
- 无明显风险时返回空 findings，并解释原因。
- 后端测试通过。

### US-007 Generate actionable review suggestions

**用户故事**：作为开发者，我希望 Review 评论能解释问题并给出修复建议，以便快速行动。

**优先级**：7

**验收标准**：

- Review 建议与具体风险发现和变更文件关联。
- 每条建议包含评论正文、原因、修复建议和是否阻塞合并。
- 避免缺少代码证据的泛泛评论。
- 针对变更行为和风险区域提供测试建议。
- 默认支持中文输出，并预留未来语言切换字段。
- 后端测试通过。

**用户故事**：作为开发者，我希望将评审分析拆成可观测的工作流节点，以便独立管理准确性、延迟和失败。

## 10. 评分标准对齐

### 10.1 作品完整度与创新性 40%

- 完整覆盖 PR 输入、代码获取、上下文分析、总结、风险识别、建议生成和模拟评论。
- 创新点体现在上下文增强、结构化风险分类、证据引用、置信度控制和 LangGraph 可观测工作流。
- 前端提供完整闭环体验，而不是单纯命令行脚本。

### 10.2 开发过程与质量 40%

- 架构分层清晰：API、Provider、Context、Workflow、LLM、Guardrail、Frontend。
- PR 粒度可控，每个 PR 聚焦一个功能。
- commit 在比赛周期内持续提交，避免最后集中导入。
- README 声明依赖、运行方式和原创边界。
- 关键逻辑包含测试，主分支保持可运行。

### 10.3 演示与表达 20%

## 13. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| GitHub API 限流或 Token 缺失 | 无法获取真实 PR | 提供 Mock Provider 和示例 PR 数据 |
| 模型输出格式不稳定 | 前端无法渲染 | 使用 Pydantic Schema、重试和降级策略 |
| 误报过多 | 用户不信任结果 | 增加置信度阈值、证据约束和自检节点 |
| 上下文过长 | 响应慢或超出模型限制 | 对文件和 hunk 设置预算，优先保留相关上下文 |
| 3 天时间紧张 | 功能不完整 | 优先完成 MVP 闭环，真实评论发布改为模拟 |
| 最终提交不符合规则 | 作品可能无效 | 使用交付清单检查 README、PR、commit、demo 视频和仓库权限 |
